## 1. 痛点：信息过载 → 收藏即坟墓

你每天都在收藏——微信文章、技术博客、Newsletter、行业动态、论文线索。但 95% 的收藏再也不会被打开。

这不是懒惰，是**结构性问题**：

- **人类消化速度 << 信息流入速度**：日接触 50-100 条有效线索，深度处理能力仅 5-10 条
- **收藏 ≠ 知识**：收藏是人类对信息过载的妥协——"先存着以后看"≈"永远不会看"
- **断点在"收藏后"**：现有工具只解决"怎么收"，不解决"收了之后怎么办"

**一句话**：**信息过载导致收藏变成坟墓，而坟墓里的知识从未被激活。**

---

## 2. 解决方案：把收藏变成 Agent 可消费的知识资产

Universal Collector 不改变你的收藏习惯（还是粘贴链接），但改变了**收藏之后发生的事**：

```
你粘贴一个链接
    ↓  <10 秒
AI 自动抓取 → 分类 → 摘要 → 结构化存储
    ↓
你的 Agent 可以：查询、复用、生成报告、转化 Skill
```

**核心范式转换**：从 **Human-Oriented**（为人浏览设计的书签夹）到 **Agent-Oriented**（为 AI 消费设计的知识管道）。

**人只需要做一件事：选择信息源。** 剩下的——抓取、分类、摘要、存储、检索、复用——全部交给 Agent。

具体来说：
- **收藏**：粘贴链接 / 一键转发，<2 步完成
- **加工**：AI 自动生成结构化知识卡片（分类、标签、摘要、重要性评分、关联行动项）
- **消费**：Agent 用自然语言查询你的知识库、注入写作上下文、生成主题报告、一键转化为可发布 Skill

## 4. 产品形态

### 三层管道架构

```
┌─────────────────────────────────────────────────────┐
│  Ingestion Layer（摄取层）                           │
│  微信 / 浏览器 / 邮件 / RSS → 一键收录，自动抓取     │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  Processing Layer（加工层）                          │
│  LLM 自动分类 + 摘要 + 标签 + 重要性评分 + 行动项    │
│  输出：结构化知识卡片（Markdown + JSON）              │
└──────────────────────┬──────────────────────────────┘
                       ↓
┌─────────────────────────────────────────────────────┐
│  Consumption Layer（消费层）                         │
│  Agent 查询 · MCP Server · 主题报告 · Skill 生成    │
└─────────────────────────────────────────────────────┘
```

### 关键技术特性

| 特性 | 说明 |
|------|------|
| **Local-First** | 数据存在本地，Markdown + JSON 开放格式，零锁定 |
| **Agent-Native** | 数据格式、元数据、接口全部为 Agent 消费优化 |
| **MCP Server** | 任何支持 MCP 的 Agent 都能把 UC 当作长期记忆层 |
| **向量化检索** | ChromaDB embedding 支持语义搜索（Phase 1） |

## 5. 差异化：为什么不是 Readwise / Cubox / Mem0？

### 市场空白地图

```
Agent 集成度（高 ↑）

    ┌──────────────────────────────────────────────┐
    │  ★ UC 目标位置        Mem0      Supermemory  │
    │  低摩擦 + 高 Agent    (高摩擦)   (高摩擦)     │
    │                                              │
    │         ← 市场空白 →                         │
    │                                              │
    │  Readwise             Quivr                  │
    │  (低摩擦, 有限Agent)   (中摩擦, 中Agent)      │
    │                                              │
    │  Karakeep  Cubox  NotebookLM  Otio           │
    │  (人类消费为主)                              │
    └──────────────────────────────────────────────┘
         低 ←────── 收藏摩擦 ──────→ 高
```

**左上象限（低摩擦 × 高 Agent 集成）完全空白——这就是 UC 的定位。**

### 差异化支柱

| 我们 | 他们 |
|------|------|
| 为 Agent 消费设计（开放格式 + 结构化元数据 + MCP 接口） | Readwise/Cubox：为人类浏览设计 |
| 终端用户友好（粘贴链接即用，零配置） | Mem0/Supermemory：仅开发者可用 |
| 开放 + 本地优先（数据主权） | NotebookLM/Otio：封闭花园 |
| 知识复用闭环（收藏 → Agent 查询 → Skill → 报告） | 所有竞品都缺少"收藏→复用"闭环 |

## 6. 商业模式

### Open-Core 三步走

| 阶段 | 时间 | 产品 | 收入 |
|------|------|------|------|
| **Phase 0** | 0-3 月 | 开源核心（CLI + 核心管道 + 开放 schema）| 社区建设，零收入 |
| **Phase 1** | 3-6 月 | ClawHub Skill 分发 + MCP Server 发布 | Skill 销售 + 捐赠（少量） |
| **Phase 2** | 6-12 月 | UC Cloud（可选云同步 + 高级 Agent 功能）| $5-8/月订阅（主力） |
| **Phase 3** | 12-24 月 | 团队版 + 企业版 | 企业合同（高客单价） |

### 为什么是 Open-Core 而不是 SaaS

| 维度 | Open-Core | SaaS |
|------|-----------|------|
| OPC 可行性 | ⭐⭐⭐⭐⭐ 零获客成本 | ⭐⭐⭐ 高 CAC |
| 数据主权一致性 | ⭐⭐⭐⭐⭐ 品牌即承诺 | ⭐⭐ "数据上云"否定差异化 |
| AI 运营杠杆 | ⭐⭐⭐⭐⭐ 社区 AI 化 | ⭐⭐⭐ 客服难自动化 |
| 长期护城河 | 生态锁定（Obsidian 路径）| 红海竞争 |

### 盈亏模型

| 月份 | 活跃用户 | 付费用户 | MRR | 月利润（¥） |
|------|---------|---------|-----|-----------|
| 6 | 1,000 | 20-100 | $120-600 | -¥330 ~ +¥2,850 |
| 12 | 3,000-5,000 | 60-250 | $360-1,500 | +¥1,110 ~ +¥8,875 |
| **盈亏平衡** | **~10-12 月** | **~60 用户** | **~$360** | — |

## 8. 团队与运营

### AI-Native OPC（One-Person Company）

| 角色 | 执行者 | 占比 |
|------|--------|------|
| 产品 + 战略 | 创始人 | 30% |
| 工程 | 创始人 + AI Agent | 40% |
| 内容 + 营销 | AI Agent | 15% |
| 社区运营 | AI Agent | 10% |
| 客户支持 | AI Agent | 5% |

**AI 替代率目标**: ~80%（内容、支持、研究、文档由 AI 完成）

### 技术栈

- **存储**: Markdown + JSON + ChromaDB（向量）
- **AI**: LLM 分类/摘要 + Embedding 检索
- **接口**: CLI → MCP Server → API
- **基础设施**: Local-First，可选 Cloud（Supabase + Vercel）

## 9. 风险与应对

| 风险 | 概率 | 应对策略 |
|------|------|---------|
| **Scope 蔓延** | 高 | Agent-Native 五维检验清单，任何功能必须通过定位检验 |
| **大厂复制** | 中 | 开源协议 + 社区生态锁定 + 12-18 月窗口期抢先发布 |
| **验证缺口** | 高 | 仅 1 用户 4 篇验证，需尽快获取 5-10 外部用户 |
| **微信反爬** | 中 | 备选方案：邮件转发收录 + RSS + 浏览器插件 |
| **Open-Core 变现慢** | 中 | 保守估计 10-12 月盈亏平衡；Skill 市场补充早期收入 |

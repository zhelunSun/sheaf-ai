# Universal Collector：产品经理视角分析

> 从「用户痛点」到「产品定义」的翻译

---

## 一、产品定位

### 一句话定义

**Universal Collector 是一个 Agent-Native 的知识收录与转化系统——你只管丢链接，Agent 自动分类、摘要、关联，并让你的收藏成为可对话、可复用、可转化为 Skill 的知识资产。**

### Elevator Pitch

> 现有的收藏工具（荣耀收藏、Readwise、印象笔记）都在解决同一个问题：**怎么把东西存好**。但它们都忽略了一个事实：收藏的真正目的不是「存」，而是「用」。
>
> Universal Collector 不做一个更好的盒子。它做一个**知识路由器**——收录端低摩擦、加工端全自动、输出端 Agent-Ready。你的每一条收藏，都会自动变成 Agent 能检索、能引用、能对话的知识节点。
>
> 区别于一众 Human-Oriented 的收藏产品，UC 从第一天就为 Agent 消费设计。

## 二、目标用户画像（泛化版）

| 维度 | 描述 |
|------|------|
| **角色** | 知识工作者：研究者、开发者、产品经理、投资人、独立创作者 |
| **共同特征** | 信息 diet 高杂、跨领域、时间碎片化、已有 Agent/AI 工具栈 |
| **技术成熟度** | 愿意使用 AI 工具，对 API/Token 成本有基本认知 |
| **核心痛点** | 收藏后吃灰、信息检索困难、知识无法被现有 Agent 复用 |
| **付费意愿** | 对「省时间」和「降低焦虑」敏感，愿为 token 和效率付费 |

## 三、核心价值主张：用 JTBD 框架拆解

> JTBD (Jobs-to-be-Done)：用户「雇佣」一个产品，是为了完成某个「任务」，而不是为了产品本身的功能。

### 3.1 用户要完成的「任务」

| 任务 | 当前解决方案 | 为什么失败 |
|------|------------|-----------|
| **当我看到一篇有价值的文章，我想确保它不会被遗忘** | 微信收藏 / 浏览器书签 | 收进去就是等死，没有后续加工 |
| **当我需要回顾某个主题的信息时，我想快速找到相关收藏** | 手动翻收藏夹 / 搜索聊天记录 | 分类混乱，无法语义检索 |
| **当我在写作/研究时，我想引用我之前看到过的观点** | 凭记忆 / 重新搜索 | 记忆不可靠，重新搜索浪费时间 |
| **当我想追踪某个领域的动态时，我想自动获得结构化总结** | 手动整理周报 / 订阅 Newsletter | 人工成本高， Newsletter 太泛 |
| **当我的 Agent 需要知识支撑时，我想让它能访问我的收藏** | 无 | 现有收藏工具没有 Agent 接口 |

### 3.2 UC 的价值主张

```
传统收藏工具          Universal Collector
    │                       │
    ▼                       ▼
「帮你存好」          「帮你用好」
    │                       │
    ▼                       ▼
Human-Oriented      Agent-Oriented
（为人眼浏览设计）    （为 Agent 消费设计）
    │                       │
    ▼                       ▼
静态归档              动态知识资产
    │                       │
    ▼                       ▼
你去找它              它来找你（主动推送）+
                      它懂你要什么（Agent 对话）
```

### 4.1 竞品定位图

擦  │                     │                     │  擦
    │   Readwise/Omnivore │                     │
    │   (阅读同步型)       │                     │

### 4.2 差异化对比表

| 维度 | 荣耀 Yoyo 收藏 | Readwise | 印象笔记/Notion | **Universal Collector** |
|------|-------------|----------|----------------|------------------------|
| **收录摩擦** | 系统级，低 | 需浏览器插件/转发邮件，中 | 需手动复制粘贴，高 | **微信转发/插件一键，低** |
| **自动分类** | 有，但分类杂 | 仅按来源分 | 无（全靠手动） | **AI Native 自动子分类** |
| **AI 总结质量** | 一般，通用模板 | 提取高亮+摘要，较好 | 依赖用户整理 | **主题感知+结构化输出** |
| **Agent 接口** | ❌ 封闭生态 | ❌ 无 | ⚠️ API 有限 | **✅ 原生 Agent-Oriented** |
| **开放扩展** | ❌ 仅荣耀设备 | ⚠️ 可导出 | ⚠️ 数据可导出 | **✅ 开放格式+可配置 API+Skill 化** |
| **交互范式** | Human 浏览 | Human 阅读 | Human 整理 | **Agent 对话 + Human 确认** |
| **知识复用** | 回看收藏夹 | 复习高亮 | 手动引用 | **Agent 直接检索+Skill 转化** |

### 4.3 核心差异化：Agent-Oriented vs Human-Oriented

这是 UC 与市面上所有收藏工具的本质区别：

| | Human-Oriented（传统） | Agent-Oriented（UC） |
|---|----------------------|---------------------|
| **设计假设** | 用户会打开收藏夹浏览 | 用户会和 Agent 对话 |
| **分类目的** | 帮助人眼快速定位 | 帮助 Agent 精准检索 |
| **摘要目的** | 帮助人快速判断要不要读原文 | 帮助 Agent 理解内容+回答用户问题 |
| **存储格式** | 富文本/PDF（为人阅读优化） | Markdown/JSON/Embedding（为 Agent 消费优化） |
| **输出形式** | 收藏夹列表 / 阅读流 | Agent 对话 / 结构化报告 / Skill 源码 |

## 六、交互范式：私人图书馆管理员

> 最好的交互不是「打开一个 App 看收藏列表」，而是「和一个懂你的 Agent 对话」。

### 6.1 用户与 Agent 的对话模式

```
用户：@知识管理员 最近我收了哪些关于多智能体的文章？

Agent：过去 7 天你收录了 4 篇相关文章：
1. [文章A] — 讲的是 Multi-Agent 的规划协调（科研主题）
2. [文章B] — 对比了 AutoGen 和 CrewAI 的设计差异（AI 技术主题）
3. [文章C] — 某创业公司用 Multi-Agent 做客服落地（AI 产品主题）
4. [文章D] — 一篇综述，提到了你之前在关注的 GraphRAG（跨主题）

需要我：
- 生成一份对比摘要？
- 把这几篇的要点更新到你的 "Multi-Agent 追踪" 知识库？
- 检测到文章D提到了一个 6 月 15 日的 workshop deadline，要加入提醒吗？
```

### 6.2 人机分工原则

| 机器做 | 人做（可选干预） | 完全自动 |
|-------|---------------|---------|
| 收录内容抓取 | 分类纠偏（如果错了） | 元数据提取 |
| 主题分类 | 摘要修正 | 子主题聚类 |
| 摘要生成 | 重要性调整 | 关联推荐 |
| 报告生成 | 报告方向定制 | 时效性识别 |
| 回答用户查询 | Skill 转化确认 | 向量检索 |

**原则**：默认全自动，用户随时可介入纠偏，但不介入也能跑。

## 七、技术实现路径

### 7.1 MVP 技术栈（Phase 0）

```
收录端：
  - 微信公众号文章：支持链接粘贴/转发（先手动，后自动化）
  - 手动 Markdown 输入：直接作为 fallback

处理端：
  - 文章抓取： requests + BeautifulSoup / playwright（防反爬）
  - 内容解析： LLM（SiliconFlow / DeepSeek，便宜模型做初筛）
  - 分类： LLM + 简单规则（四大主题先硬编码，后续让 LLM 自动演化）
  - 摘要： LLM 一次性生成
  - 存储： Markdown 文件 + JSON 元数据（本地文件系统）

消费端：
  - Agent 对话： WorkBuddy 现有聊天界面
  - 检索： 简单关键词搜索（先不做向量检索）
  - 报告： LLM 按需生成
```

### 7.2 Phase 1（增强）

```
- 接入微信文章自动化（可能需要企业微信/公众号接口或 RPA）
- Embedding 向量存储（ChromaDB / LanceDB，复用 nova-reader 经验）
- 语义检索（向量+关键词混合）
- 浏览器插件（一键收录）
- 定时任务（WorkBuddy automation 对接）
```

### 7.3 Phase 2（生态）

```
- Skill 一键转化（对接 skill-factory）
- 与 nova-reader 论文精读联动
- 知识图谱（轻量版，同主题文章关联）
- 多用户支持（如果要产品化）
```

## 八、风险与假设

### 8.1 关键假设

| 假设 | 验证方式 | 风险等级 |
|------|---------|---------|
| 用户愿意为「AI 自动整理收藏」付出 Token 成本 | MVP 试用观察 | 低 |
| 微信文章链接可以被稳定抓取（防反爬） | 技术预研 | **中** |
| AI 自动分类准确率可达可用水平（>80%） | MVP 小规模测试 | **中** |
| Agent 对话式检索比传统收藏夹更高效 | 用户反馈 | 低 |
| 用户愿意接受「全自动+可纠偏」而非「全手动」 | 产品试用 | 低 |

### 8.2 主要风险

1. **微信生态封闭风险**：微信文章抓取可能受反爬限制，需要探索稳定方案（企业微信接口、RPA、或用户手动粘贴）
2. **分类质量风险**：初期 LLM 分类可能不准，需要快速迭代 prompt + 允许用户纠偏
3. **信息过载反噬**：如果系统推送太多，可能从「收藏焦虑」变成「整理焦虑」，需要重要性过滤
4. **Scope 蔓延风险**：用户可能希望「什么都能收」，需要守住 MVP 边界（先微信文章，后扩展）
5. **与现有工具竞争**：Notion/Readwise 已有忠实用户，UC 需要明确的 Agent-Oriented 差异化

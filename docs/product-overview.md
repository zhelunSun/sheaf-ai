# Universal Collector 产品概览

> Agent-Native 的知识收录与转化系统

## 产品定位

**一句话定义**：Universal Collector 是一条「你丢链接 → AI 自动分类摘要 → Agent 可对话查询」的知识转化流水线。

**Elevator Pitch**：现有的收藏工具（荣耀收藏、Readwise、印象笔记）都在解决「怎么把东西存好」。但收藏的真正目的不是「存」，而是「用」。UC 不做一个更好的盒子，它做一个**知识路由器**——收录端低摩擦、加工端全自动、输出端 Agent-Ready。区别于一众 Human-Oriented 的收藏产品，UC 从第一天就为 Agent 消费设计。

## 大愿景：AI 时代的个人知识秘书

AI 生成内容正在变得极度廉价。人类的处理带宽不会增长。这两条线交叉的结果是：

> **信息的价值不再在于「获取」，而在于「筛选和消化」。**

用户不需要更多内容源，不需要更强的搜索引擎。用户需要的是一个**私人知识秘书**——从用户信任的信息源中，自动抽取知识，组织成可复用的个人知识资产，在用户需要时精准呈现。

Universal Collector 要做的就是这位秘书的**基础设施层**：
- **输入**：用户选择信息来源（微信、Twitter、RSS、邮件）
- **处理**：AI 自动分类、摘要、打标签、识别时效性
- **输出**：结构化知识资产，Agent 可查询、可复用、可转化为 Skill

核心转变：**从「人找信息」到「信息找人」**。用户只负责挑选信息源，消化、整理、检索、复用全部交给 Agent。

## 核心差异化：Agent-Oriented vs Human-Oriented

| 维度 | Human-Oriented（传统） | Agent-Oriented（UC） |
|------|----------------------|---------------------|
| **设计假设** | 用户会打开收藏夹浏览 | 用户会和 Agent 对话 |
| **分类目的** | 帮人眼快速定位 | 帮 Agent 精准检索 |
| **摘要目的** | 帮人快速判断是否读原文 | 帮 Agent 理解内容+回答用户问题 |
| **存储格式** | 富文本/PDF（为人阅读优化） | Markdown/JSON/Embedding（为 Agent 消费优化） |
| **输出形式** | 收藏夹列表 / 阅读流 | Agent 对话 / 结构化报告 / Skill 源码 |
| **交互方式** | 打开 App → 翻分类 → 翻列表 | 直接问："最近关于 RAG 的收藏？" |

## 技术路线：提示词分类 vs Embedding

### 当前方案（MVP）：提示词 + 大模型分类

- 用 LLM（DeepSeek-V3.2）直接对文章内容做分类、标签、摘要
- 成本极低（~¥0.3/月），准确率高（大模型理解力强）
- 已经跑通 4/4 文章端到端验证

### Embedding 方案：何时引入，为什么？

**结论：Embedding 不是护城河，也不是当前的优先级。**

| 维度 | 提示词分类（当前） | Embedding 检索（远期） |
|------|-------------------|---------------------|
| **准确率** | 高（大模型直接理解语义） | 中高（向量相似度 ≠ 语义理解） |
| **成本** | 极低（¥0.3/月） | 中（embedding 计算 + 向量库维护） |
| **强项** | 分类、摘要、标签、时效性识别 | 语义搜索、相似文章发现、聚类 |
| **弱项** | 无法做模糊语义检索 | 分类和摘要仍需要 LLM |
| **本质** | **理解**（NLU） | **检索**（NIR） |

**Embedding 的真正价值在于**：
1. 当收藏量超过 100 条，关键词检索不够用时，提供语义检索能力
2. 发现「收藏之间的隐含关联」（如两篇不同标签的文章讨论同一个底层技术）
3. 支持模糊查询（如「我之前收藏过一篇讲 Agent 记忆的文章，忘了标题」）

**但 Embedding 不是独立价值**——它只是检索增强手段，不是分类替代品。当前阶段（4 条收藏、手动粘贴链接），引入 Embedding 是过度工程。

**引入时机**：收藏量超过 50-100 条、关键词检索明显不够用的时候。

## 三层架构

**Layer 1 — 收录层（Ingestion）**
- P0（当前 MVP）：用户手动粘贴链接
- P1：浏览器插件 / 桌面悬浮球（拖拽文件/URL）
- P2：微信自动化（RPA/企业微信接口）

**Layer 2 — 加工层（Processing），用户只管丢，AI 负责理**
- 文章抓取：requests → playwright → 用户粘贴正文（三级 fallback）
- LLM 理解（DeepSeek-V3.2，~¥0.3/月）
  - 主题分类：科研 / 市场投资 / AI 产品 / AI 技术
  - 子主题：LLM 自动提取（不预设分类树）
  - 标签：自动提取，支持跨主题关联
  - 摘要：一句话 takeaway + 结构化要点
  - 时效性识别：deadline / CFP / 发布日期自动提取
- 存储：Markdown 原文 + JSON 元数据 + index.jsonl 全局索引

**Layer 3 — 消费层（Consumption），Agent 是交互接口**
- P0：WorkBuddy Agent 对话式检索
- P1：定时报告（自动化）、Embedding 语义检索（50+ 收藏后引入）
- P2：Skill 一键转化、论文联动（nova-reader 插件）

## 交互范式：私人知识秘书

```
你：最近我收了哪些关于 RAG 的文章？
Agent：过去 7 天 3 篇...
        需要生成对比摘要吗？检测到 6.15 deadline，要提醒吗？
```

## 非功能约束

| 维度 | 要求 |
|------|------|
| **开放** | Markdown/JSON 格式，数据用户掌控，随时可导出 |
| **本地优先** | 数据存在本地文件系统，无 SaaS 锁定 |
| **成本敏感** | DeepSeek-V3.2 主力，~¥0.3/月 |
| **增量建设** | 从微信文章开始，逐步扩展源和功能 |
| **Agent-Oriented** | Agent 是用户与知识库交互的主要接口 |
| **痛点驱动** | 不追求技术先进性，追求解决用户真实痛点 |

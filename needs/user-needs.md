# 全局收藏系统：用户需求文档（User Needs Doc）

> 视角：作为终端用户，我（zhelunSun）到底需要什么？

---

## 一、我是谁：用户画像

> **泛化定位**：本产品面向所有「从事知识密集型工作、信息 diet 高杂、时间碎片化」的人群——研究者、开发者、投资人、产品经理、独立创作者等。以下以 zhelunSun 为典型用户代表。

| 维度 | 描述 |
|------|------|
| **身份** | 清华大学博士生（AI + 遥感交叉），同时关注 AI / Web3 投资；Skill Builder / AI Agent Tinkerer |
| **信息 diet** | 高密度、跨领域、中英文混合；日均接触 50-100 条有效信息线索 |
| **工具栈** | WorkBuddy / ClawHub / 微信 / 浏览器 / 邮件 / 会议通知 / 各类 SaaS |
| **时间特征** | 碎片化阅读多（地铁上、吃饭时），深度处理时间少（需要 AI 代劳）；愿意为「省时间」付费 token |
| **核心诉求** | 别让有价值的信息死在收藏夹里；让收藏成为 Agent 的知识资产，而非个人记忆负担 |

## 二、我现在怎么「收藏」？

### 2.1 典型场景（带情绪的真实记录）

#### 场景 A：微信公众号与技术文章（最高频）
> 早上通勤刷到几篇微信公众号文章——一篇讲新的 Agent 框架设计，一篇分析某大模型公司的融资，一篇遥感领域的最新进展。中午吃饭时又看到朋友转了一篇 a16z 博客的中文翻译。

- **现在怎么做**：觉得有价值的点「收藏」或「转发到文件传输助手」，想深度读的复制链接丢进临时 markdown。
- **之后发生什么**：微信收藏夹 500+ 条从未回顾，文件传输助手沦为信息坟场，临时 markdown 三个月后不知所踪。
- **情绪**：焦虑。明知道里面有很多有价值的判断和线索，但永远没时间整理。

#### 场景 A'：科研论文与会议（低频，但重要）
> 偶尔在群里看到一篇 arXiv 论文链接，或导师转的会议 deadline、workshop CFP。

- **现在怎么做**：arXiv 高质量论文走 nova-reader 精读流程；会议/CFP 等轻量信息希望被 collector 自动识别并提醒。
- **边界判断**：完整 PDF 论文 ≠ collector 核心场景，collector 收「论文链接/公众号总结/轻量线索」，深度精读留给 nova-reader（未来可作为扩展插件对接）。
- **情绪**：希望有分层——快速线索别丢，但别和深度阅读混为一谈。

#### 场景 B：市场与投资信息
> 看到一条 Twitter/X 上说某个 AI infra 公司融资了，或者某个 Web3 项目的 TVL 数据变化，或者一篇 a16z 的博客。

- **现在怎么做**：截图发微信收藏，或者浏览器书签新建文件夹「AI/投资」。
- **之后发生什么**：微信收藏是个黑洞；浏览器书签从 2024 年开始就没整理过。
- **情绪**：无力。信息看了但没形成判断，想复盘的时候找不到依据。

#### 场景 C：AI 产品与技术动态
> 一个新的 LLM 发布（如 Gemini 2.5 Pro），一个新的开源框架（如某 Agent 框架），一个新的评测基准。

- **现在怎么做**：快速浏览，觉得重要就丢进 `ai-science-agent/` 或某个临时 markdown。
- **之后发生什么**：半年后发现这个框架已经过时，但当时的关键判断（「这个比 AutoGen 好在哪」）已经忘光了。
- **情绪**：可惜。当时的直觉和判断没有留存。

#### 场景 D：通知与机会
> 学校的一个交叉学科基金申请通知，一个暑校/冬校的招生，一个开源之夏的导师招募。

- **现在怎么做**：邮件标星，日历上随便设个提醒。
- **之后发生什么**：提醒响的时候已经过期，或者根本不记得当初为什么觉得重要。
- **情绪**：后悔。机会成本难以估量。

### 2.2 当前工具矩阵与痛点

| 内容类型 | 当前工具 | 痛点 |
|---------|---------|------|
| 论文 | `interested_papers.md` / nova-reader | 论文有专门流程，但「轻量相关论文」和「会议/ workshop」进不来 |
| 网页/文章 | 浏览器书签 / 微信收藏 | 没有分类体系，无法检索，无法自动摘要 |
| 邮件通知 | Gmail 标星 | 与其他信息源割裂，没有上下文 |
| 社交媒体 | 截图 / 转发给自己 | 纯文本丢失，无法结构化 |
| 会议/活动链接 | 微信文件传输助手 | 完全不可管理 |

### 2.3 核心痛点总结

```
收藏前：多入口（微信/浏览器/邮件/会议）→ 信息分散
    ↓
收藏时：无统一分类 → 靠记忆和直觉乱放
    ↓
收藏后：无自动处理 → 堆积成山
    ↓
需要时：找不到 / 想不起 / 用不了
    ↓
结果：重复收藏、信息焦虑、知识浪费
```

## 三、我想要什么？

### 3.1 理想工作流（用户口述版）

```
看到东西 → 一键/低摩擦收录 → 自动分类+摘要 → 
（可选）我花 30 秒确认/打标签 → 
进入知识库 → 可被 Agent 调用 / 可生成 Skill / 可输出周报
```

### 3.2 功能需求（按优先级）

#### P0：收录与分类
- [ ] **微信文章优先接入**：支持链接收录（先跑通微信生态，其他源后续扩展）
- [ ] **AI Native 自动分类**：用户只管「丢进来」，系统自动分到主题 + 子主题；支持四大主题：科研 / 市场投资 / AI 产品 / AI 技术
- [ ] **自动元数据提取**：标题、来源、时间、作者、关键句/摘要
- [ ] **低摩擦收录**：收录动作不超过 2 步（如转发到某个地方，或浏览器插件一键）
- [ ] **论文边界处理**：轻量论文线索（链接/公众号总结）纳入 collector；完整 PDF 深度精读未来对接 nova-reader（扩展插件）

#### P1：加工与转化
- [ ] **自动摘要**：生成 1-3 句话的核心 takeaway
- [ ] **标签/关联**：自动提标签，支持跨主题关联（如一篇论文既属于「科研」又涉及「AI 技术」）
- [ ] **重要性打分**：AI 根据我的历史行为和主题偏好，给内容打「高/中/低」优先级
- [ ] **过期提醒**：会议 deadline、申请截止日期等可自动识别并提醒

#### P2：输出与复用
- [ ] **Agent 交互式查询**：像和 Claw 聊天一样，和「掌管知识的 Agent」对话——「最近我收了哪些关于多智能体的文章？」「对比一下我收藏的这两篇框架设计」
- [ ] **定时报告（可配置）**：支持用户手动设置频率（实时/每日/每周），像 WorkBuddy 定时任务一样推送收藏洞察
- [ ] **Agent 可用**：收藏的内容能被 WorkBuddy Agent 检索和引用（如写论文时引用某篇收藏的文章）
- [ ] **Skill 转化**：某个主题下积累足够内容后，可一键生成/更新一个 Skill（如「AI 产品动态追踪」Skill）
- [ ] **自然语言搜索**：支持语义搜索收藏历史（如「去年我收藏过哪些关于 GraphRAG 的文章？」）

### 3.3 非功能需求

| 维度 | 要求 |
|------|------|
| **开放** | 数据格式开放（Markdown/JSON），我可以随时导出；不锁定在某一平台 |
| **本地优先** | 数据存在本地（或我控制的云端），不是某个 SaaS 的黑盒 |
| **成本敏感** | Token 消耗可控，优先用便宜模型做粗处理，贵模型做精处理 |
| **增量建设** | 不需要一次搭建完美系统，可以从一个主题（如「科研」）开始，逐步扩展 |
| **与现有工作流融合** | 不取代 nova-reader / thesis-exec / skill-factory，而是补充和连接它们 |
| **Agent-Oriented** | Agent 是用户交互的主要接口，系统天然为 Agent 消费设计，而非仅面向人类浏览 |

## 四、什么是我**不想要**的？

1. **另一个 Readwise / 印象笔记**：我已经试过了，收进去就是等死。我需要的是**自动加工**和**主动输出**，不是更好的收纳盒。
2. **全自动化到失控**：我需要在关键节点（如分类是否正确、摘要是否准确）有确认权，不能完全黑盒。
3. **需要我手动维护复杂分类体系**：分类应该自动演化，我可以纠偏，但不应该从零搭建 ontology。
4. **孤立系统**：如果收藏的东西不能被我的 Agent 调用、不能转化为 Skill、不能输出结构化报告，那和浏览器书签没有本质区别。
5. **过度工程化**：我不需要一个面向百万用户的 SaaS，我需要一个**服务于我的 Agent 生态的知识管道**。

## 五、成功标准（怎么算「好用」？）

| 指标 | 当前状态 | 目标状态 |
|------|---------|---------|
| 收藏后 7 天内被回顾/处理的比例 | ~5% | >50% |
| 找一个「我上个月收藏的某篇文章」所需时间 | >5 分钟（经常找不到） | <30 秒 |
| 收藏内容被 Agent/Skill 复用的比例 | ~0% | >30% |
| 每周花在信息整理上的时间 | ~2 小时（焦虑性整理） | <15 分钟（确认性干预） |
| 因信息遗漏导致的后悔事件 | 每月 1-2 次 | 每季度 <1 次 |

## 七、已澄清的关键决策

1. **收录入口**：**微信公众号文章是最高频来源**，以链接形式呈现。论文内容管理难度大，优先级让位于微信文章。完整 PDF 论文深度精读未来对接 nova-reader（扩展插件）。

2. **分类方式**：**AI Native 自动子分类**，用户只管拖东西进来，Agent 自动整理好。无需用户手动维护分类体系。

3. **与 nova-reader 的边界**：collector 收「论文链接/公众号总结/轻量线索」等一般技术知识文本；完整 PDF 深度精读走 nova-reader。未来 collector 可判断「这是高质量论文」并触发 nova-reader 插件。

4. **荣耀的具体限制**：
   - **生态封闭**：只支持荣耀设备，非荣耀设备（如 PC）收不到手机推送
   - **无法扩展**：不能配置自己的 API key、不能接入 Skill、不能对接开放生态
   - **知识管理弱**：收藏内容过于杂乱，分类不清晰，AI 总结和问答输出效果不够好
   - **产品哲学差异**：荣耀是 **Human-Oriented**（为人浏览设计），我们要做 **Agent-Oriented**（为 Agent 消费设计）

5. **输出频率与交互范式**：
   - 定时报告频率由用户手动设置（像 WorkBuddy 定时任务）
   - **最佳交互方式**：像和 Claw 聊天一样，和「掌管知识的 Agent」对话——私人图书馆管理员模式

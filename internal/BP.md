## 0. 核心论点

张一鸣重新定义了信息的分发方式。Jobs 重新定义了人机交互。

Sheaf 要回答的问题是：**当 AI agent 成为知识的主要消费者，"知识"本身该是什么形态？**

前 AI 时代，知识是文章、论文、书籍——给人看的，存在人的格式里。AI-native 时代，知识需要变成机器可读的结构化单元，带溯源、可组合、可交易。Sheaf 做的就是这件事：从个人知识采集到全球知识市场的基础设施。

**Sheaf 不是 AI 知识助手，不是知识库，不是笔记软件。** 这些是大厂在做的事。Sheaf 首先是一个**极低摩擦的收藏工具**——有什么信息用户都想收藏，只需动一动手指。然后才是 Agent 上下文增厚器，最终成为 Agent Native 的知识基础设施。

---

## 1. 问题：知识的三个断裂

一个知识工作者每天接触 50-100 条信息信号——文章、论文、市场洞察、产品发布、机会提醒。把有价值的存下来，然后 **95% 再也不会被打开。**

这不是自律问题，是工具问题。现有的知识存储工具是为归档设计的，不是为思考设计的。它们存链接和文本，但不理解、不连接、不激活你存的东西。

具体的痛：

| 痛点 | 原因 | 代价 |
|------|------|------|
| 同一个洞察存了三遍，忘了已经存过 | 没有系统把新输入和已有知识关联 | 重复劳动 |
| 三个月前读过相关内容，需要时找不出来 | 存储按时间排列，不按概念 | 关键时刻掉链子 |
| AI 助手每次从零开始，无视你积累的一切 | 你的知识和你的 agent 之间没有桥梁 | AI 永远学不会你的领域 |
| 每周花 2+ 小时手动整理信息，还是不好用 | 整理工具消耗人力但产出递减 | 深度工作时间被偷走 |

**核心洞察**：现有工具在优化"存储和检索"——怎么归档、怎么找到。但在 AI 时代，问题不是找到信息，是 **让你积累的知识变得可计算**——让 agent 能推理、连接、自主行动。

## 2. 解决方案：给 AI 时代的个人知识层

Sheaf 不是又一个稍后阅读工具。它是一个知识基础设施层，放在你消费的信息和你使用的 AI agent 之间。

**范式转换**：从"存了再找"到"存了就用"。

```
你的操作：  粘贴一个链接。结束。
Sheaf 的处理：  提取 → 结构化 → 分类 → 关联 → 存为 agent 可用的资产
你的 agent 得到：  一个可查询、语义索引的知识库，理解你的领域
```

### 和现有方案的差异

| 维度 | 传统书签工具 | RAG / 向量数据库 | AI 记忆 (Mem0 等) | **Sheaf** |
|------|------------|----------------|------------------|-----------|
| 设计目标 | 人类浏览 | 开发者管线 | 对话上下文 | **Agent 消费你策展的知识** |
| 输入摩擦 | 低（一键收藏） | 高（ETL 管线） | 无（自动捕获） | **低（粘贴链接）** |
| 输出格式 | 原始文本/链接 | 向量/chunks | 对话片段 | **带溯源的结构化知识卡片** |
| 数据所有权 | 平台（通常） | 你（如果你自己搭） | 服务商 | **你——本地优先，开放格式** |
| 知识复用 | 手动复制粘贴 | 定制集成 | 限于同平台 | **任何 agent，通过 MCP/API/Skill** |

### 知识卡片：为什么结构化很重要

核心创新不是更好的搜索，是 **结构化知识资产**。

知识卡片不是摘要。它是一个 **机器可读的知识单元**：

- 从来源中 **提取主张**，不是改写
- 每个主张带 **溯源**——追溯到具体 URL
- 自动 **分类**——主题、内容类型、时效性
- 嵌入 **向量**——用于语义检索
- **置信度**——提取质量评分

当 agent 基于知识卡片推理时，它可以引用来源、检测矛盾、发现你忽略的模式、积累领域专长。原始文本和向量做不到这些。

### 端到端管线：从采集到交易

```
Phase 1: COLLECT（低摩擦，人触发）
─────────────────────────────────────────────
  粘贴链接 / 转发微信文章 / 分享 AI 对话 → Sheaf 提取
  → 条目存入 sheaf data/（原始数据）
  摩擦：< 2 秒。无文件夹。无标签。只管存。

Phase 2: CRYSTALLIZE（AI 驱动，批处理）
─────────────────────────────────────────────
  积累足够多相关条目 → sheaf crystallize "topic"
  → LLM 提取主张、模式、反模式、缺口
  → 知识卡片生成（sheaf_cards/）
  → 嵌入向量计算（numpy 余弦相似度）
  → 结晶报告（Markdown）供人审查

Phase 3: BUNDLE（创作者打包，一条命令）
─────────────────────────────────────────────
  sheaf bundle --topic "城市森林遥感"
  → 卡片 + 向量 + 溯源 + 摘要 → .sheaf.zip
  → manifest.json 定义元数据（标题、作者、版本、协议、标签）
  → 一个可移植文件：知识的 "sheaf"——一捆知识

Phase 4: PUBLISH（市场或点对点分享）
─────────────────────────────────────────────
  免费分享：sheaf publish --free（公共注册表，不收费）
  付费 bundle：sheaf publish --price 19.99（创作者定价，平台抽 15-25%）
  点对点：直接发送 .sheaf.zip（邮件、GitHub Release、U 盘）

Phase 5: CONSUME（Agent 或人读取 bundle）
─────────────────────────────────────────────
  sheaf import bundle.zip          → 卡片合并到本地知识库
  Agent via MCP Server             → 查询导入的卡片作为上下文
  sheaf info bundle.zip            → 人读 summary.md
  引用导出                          → 卡片变成论文/报告中的引用
```

**例子：一个遥感研究者的真实路径（Dogfood）**

1. **Collect**：6 个月，Sir 存了 60+ 篇城市森林遥感相关的论文、技术报告、政策文件
2. **Crystallize**：`sheaf crystallize "urban forest"` 处理 63 个来源 → 提取 42 张知识卡片，覆盖方法（LiDAR、高光谱、SAR）、物种识别、冠层指标、政策标准
3. **Bundle**：`sheaf bundle --topic "RS for Urban Forest"` → 生成 `urban-forest-rs.sheaf.zip`（42 张卡片 + 向量 + 完整溯源）
4. **Publish**：上架 Marketplace，定价 $19.99——遥感领域第一个商业化知识 bundle
5. **Consume**：巴西一个博士生导入 bundle → 他的 AI agent 有了城市森林遥感方法的结构化知识 → 研究规划质量可测量地提升

飞轮：创作者用 Sheaf → 产出有价值的 bundle → 买家发现 → 他们也开始用 Sheaf → 变成创作者 → 更多 bundle → 网络效应累积。

## 3. 市场机会

### 市场背景

三个趋势正在汇合，创造了 18 个月前不存在的窗口：

1. **AI agent 正在成为主要的知识工作者。** 2026 年，数百万专业人士每天使用 AI agent（Claude、GPT、Cursor、Copilot）。这些 agent 很强，但无状态——每次对话从零开始。记住你知道什么的 agent，价值大一个数量级。

2. **"存了就忘"的问题在恶化。** AI 时代信息速度增加了 10 倍。更多工具、更多文章、更多产品发布、更多论文。"你接触到的"和"你实际能用上的"之间的鸿沟在扩大。

3. **MCP（Model Context Protocol）标准化了 agent-记忆集成。** Anthropic 的 MCP，现已被 OpenAI、Google、Microsoft 支持，给任何工具提供标准方式将知识注入任何 agent。相当于 agent 时代的 RSS——让个人知识层变得即插即用。

### 竞争格局

没有现有产品占据 Sheaf 的目标位置：

**国内竞品补充**：ima.copilot（腾讯）、飞书知识库、语雀、Coze 等大厂产品都在做"AI 知识库"，但它们的共同点是**封闭生态**——知识锁在平台内，不做 Agent Context，不做开放协议。Sheaf 的差异化是**开放 + Agent-Native**。

### Sheaf 填补的三个结构性缺口

| 缺口 | 现状 | 影响 |
|------|------|------|
| 保存到 agent 消费的最后一公里 | 没有工具连接"用户存链接"到"agent 查询结构化知识"。RAG 需要工程。书签工具只给人看。 | 个人 AI 的基础设施缺失 |
| 结构化知识资产 vs 原始文本 | 现有方案存文本块或对话片段。没有产出 **可溯源的、主张级别的知识单元**。 | 原始检索 ≠ 理解。结构化资产才能引用、检测矛盾、复合学习 |
| 开放且易用 | NotebookLM 强但封闭。Mem0 开放但只面向开发者。没有产品同时 **开源** 和 **终端用户友好**。 | 数据主权不是哲学，是品类定义 |

## 5. 商业模式：Open-Core

### 为什么是 Open-Core，不是 SaaS

Sheaf 的价值主张是"你的知识，你拥有，任何 agent 可用"。这个理念就是产品本身。一个把你的知识上传到云端服务器的 SaaS，和核心承诺矛盾。

Open-Core 不只是定价策略——是架构和理念的统一：

| 因素 | Open-Core（Sheaf） | SaaS 免费增值 |
|------|-------------------|--------------|
| 理念一致性 | 数据主权 = 品牌承诺 | "上传到我们的云" = 信任矛盾 |
| 一人公司可行性 | 社区驱动，CAC 接近零 | 付费获客（$50-200/用户），OPC 扛不住 |
| AI 杠杆 | 社区帮你 AI 化和扩展 | 中央团队包打一切 |
| 长期护城河 | 生态绑定（Obsidian 模式） | 红海功能竞争 |

**中国因素**：SaaS 在中国市场历史上很难——用户习惯免费工具，企业销售周期漫长，国内巨头可以快速复制功能。但开源有不同逻辑：开发者信任社区驱动的工具，"免费核心→付费企业"的转化路径在全球已被验证（GitLab、Elastic、MongoDB）。**押注开源就是押注非对称上行**——下行有限（工具本身就有用），上行巨大（生态爆发）。

### 三层架构（Obsidian 模式）

### 生态治理：参考 Hugging Face、GitHub、OpenClaw

一个关键设计决策：Sheaf 生态怎么治理？

| 模式 | 参考 | 运作方式 | 治理主体 | 收入 |
|------|------|---------|---------|------|
| 公司主导 open-core | Hugging Face | 营利公司建平台；核心开源；企业版商业 | 3 联合创始人 + VC 董事会 | Enterprise Hub + Inference Endpoints + Pro 订阅 |
| 被收购的平台 | GitHub（→ Microsoft） | 独立公司被大厂收购；开源社区 + 专有功能（Copilot） | 母公司定战略 | Team/Enterprise 订阅 + Copilot |
| 社区优先框架 | OpenClaw | 开源框架 + 市场（ClawHub）；社区贡献 skills | 项目维护者 | Marketplace + 高级功能（早期） |

**Sheaf 的治理路线（渐进式）**：

| 阶段 | 治理 | 理由 |
|------|------|------|
| Alpha → Beta（2026） | 一人公司（OPC）+ 社区 | 最小可行治理——快决策，一人负责 |
| 增长期（2027） | 营利 OPC + 社区顾问委员会 | 加入结构化反馈；公司仍主导产品 |
| 规模化（2028+） | 考虑 .sheaf 格式基金会（类似 Apache） | 格式规范移交给中立机构；公司运营 Marketplace + Enterprise |

Hugging Face 是纯营利公司（$20B 估值，约 300 人，没有独立基金会），成功维持了社区信任，因为：核心平台保持真正免费开放，67% 人力投入工程/研究而非销售，创始人控制保证产品决策和社区价值观一致。

Sheaf 采取同样路径：OPC 是法律实体，社区信任靠行动赢得，.sheaf 格式以 Apache 2.0 开源——即使公司不在，生态也能存活。

## 6. 团队：AI-native 一人公司

**1 个创始人 + N 个 AI agent。人投入 10-17 小时/周，产出等价于 3-5 个全职。**

这不是理论模型——Sheaf 就是这样建出来的。每行代码、每份产品文档、每次战略分析，都通过人机协作完成，配有严格的阶段门控。

| 职能 | 人负责 | AI agent 负责 |
|------|--------|--------------|
| 产品 | 愿景、优先级、用户访谈 | 竞品调研、PRD 草稿、反馈分析 |
| 工程 | 架构、代码审查 | 开发、测试、文档 |
| 内容 | 品牌调性、核心发布 | 博客、社交媒体、通讯 |
| 研究 | 假设、主张验证 | 文献综述、数据分析 |
| 运营 | 战略决策 | Issue 分拣、社区管理 |

### 独特优势：学术-产品协同

Sheaf 的知识卡片引擎和创始人在清华大学的博士研究共享理论基础——具体说，是 **受证据治理的自主 agent 知识资产**。

这意味着：

- 核心卡片引擎在两个场景中验证：严格的学术环境（G1-G4 对照组实验，测量规划质量提升）和产品环境（低摩擦自动生成）
- 学术可信度为纯商业产品缺乏的信任信号提供了背书
- 研究-产品反馈循环双向加速：产品提出研究问题，研究巩固产品基础

```
sheaf_cards/（共享引擎）
    ├── Sheaf 产品（面向用户，简化，低摩擦）
    └── 博士论文（面向学术，严格证据治理）
```

## 7. 愿景：从工具到协议到生态

| 阶段 | Sheaf 变成 | 跨越 | 时间线 |
|------|-----------|------|--------|
| **工具** ← 现在在这里 | 更好的个人知识工具 | 一键保存→结构化卡片→agent 查询 | 现在-6 月 |
| **基础设施** | 个人知识 OS | 结晶引擎，bundle 格式，MCP 集成，多源采集 | 6-18 月 |
| **市场** | 知识交易网络 | Sheaf Marketplace：创作者卖 bundle，agent 消费，平台抽成 | 12-24 月 |
| **协议** | AI-native 知识标准 | .sheaf 格式成为 agent 消费知识的默认格式（像 .pdf 之于文档） | 24-36 月 |
| **Web3 生态** | 去中心化知识经济 | 链上溯源，token 化 bundle，创作者所有权，IP 保护 | 36 月+ |

**长期赌注**：随着 AI agent 成为知识工作的主要界面，竞争优势从"谁的 agent 最好"转向"谁的 agent 拥有最丰富的领域知识"。Sheaf 在建的基础设施层，让个人知识成为护城河。

### Web3 愿景：去中心化知识经济

知识行业最大的未解问题是 **知识产权**。从论文中提取的知识卡片归谁？怎么证明溯源？创作者怎么不经中间商获得报酬？

Web3 提供了天然答案：

| 问题 | Web3 方案 |
|------|----------|
| **IP 保护** | 知识卡片→链上哈希→不可篡改的溯源记录 |
| **创作者所有权** | Bundle 铸造为类 NFT 资产→创作者保留所有权 + 转售版税 |
| **数据主权** | 去中心化存储（IPFS/Arweave）→没有单一平台控制知识 |
| **无信任交易** | 智能合约处理支付→点对点知识销售无需中间商 |
| **防抄袭** | 链上时间戳 + 内容哈希→谁在什么时间创建了什么的密码学证明 |

Web3 为什么特别适合 Sheaf：

- 知识 bundle 是 **有明确所有权的数字资产**——token 化的天然用例
- .sheaf.zip 格式已经有 **溯源追踪**——加链上验证是扩展，不是重建
- 智能合约处理创作者报酬消除了信任问题（平台不能单方面改抽成比例）
- 去中心化和 Sheaf 的 **本地优先 DNA** 一致——你的知识不活在任何人的服务器上

Web3 路线图：
```
Phase 1（现在）：         中心化但开放——.sheaf 格式 + CLI + Marketplace
Phase 2（2027）：         混合——高级 bundle 可选链上溯源验证
Phase 3（2028+）：        去中心化——L2 上的知识交易网络（Base/Arbitrum）
                         创作者 token，bundle NFT，去中心化注册表
```

### Web3 竞品格局（2026-05-23）

| 协议 | 定位 | 融资 | 与 Sheaf 的关系 |
|------|------|------|----------------|
| **Story Protocol** | IP 管理 L1 区块链 | $140M（a16z） | 最强潜在合作伙伴——做 IP 基础设施，Sheaf 做知识应用层 |
| **Ocean Protocol** | 去中心化数据市场 | 成熟 | 互补——Ocean 做数据交易，Sheaf 做知识（提炼物）交易 |
| **Lens Protocol** | 去中心化社交图谱 | Aave 团队 | 轻度关联——知识市场的"发现"和"推荐"可借鉴 |
| **Arweave** | 永久区块链存储 | 成熟 | 底层存储方案——知识包可选 IPFS（临时）或 Arweave（永久） |

**推荐策略**：先集成 Story Protocol 的许可框架（MVP），规模化后考虑独立构建。Story 做基础设施（IP 注册 + 许可 + 版税），Sheaf 做应用层（知识包生产 + 消费），完美互补。

这是长期愿景，不是当前优先级。核心产品必须先证明 PMF。但架构（开放格式 + 溯源 + 创作者经济）从第一天就为 Web3 兼容设计。
